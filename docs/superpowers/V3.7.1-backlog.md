# PetForge V3.7.1 - Quota follow-ups

Items found during V3.7 quota live testing (2026-05-20 / 21). Non-blocking
for the V3.7.0 release but should ship before promoting the feature broadly.

## 1. `UV_HANDLE_CLOSING` libuv warning on CLI exit

**Symptom**: Running `petforge quota` or `petforge quota enable` on Windows
prints to stderr right after the normal output:

```
Assertion failed: !(handle->flags & UV_HANDLE_CLOSING), file src\win\async.c, line 76
```

The command itself succeeds, the state is written, exit code is 0. Pure
cosmetic noise. Not seen on the daemon path because the daemon never exits.

**Probable cause**: Node 20's built-in `fetch` (undici) keeps the global
dispatcher's connection pool alive after the single probe. When `main()`
resolves and `process.exit(code)` runs, libuv still has an async handle in
CLOSING state for an undici keep-alive socket.

**Suggested fix**:
1. After every quota CLI command, explicitly close the undici global
   dispatcher before returning. Either `import { getGlobalDispatcher } from "undici"`
   (undici is bundled with Node 20+ but not exposed by default - may need
   `node --experimental-global-undici-default-dispatcher` flag) or use a
   per-call dispatcher with `keepalive: false`.
2. Easier alternative: replace `process.exit(code)` in `src/index.ts` with
   `process.exitCode = code`, then let Node exit naturally once all handles
   close. Acceptable as long as no command leaks handles indefinitely.
3. Hacky-but-bounded alternative: `await new Promise(r => setTimeout(r, 50))`
   after the probe returns, then exit. The 50ms gives undici time to flush.

Pick (1) or (2). Add a test that captures stderr after a fixture probe and
asserts it does not contain `UV_HANDLE_CLOSING`.

## 2. `petforge quota` (no arg) is read-only, contrary to spec

**Symptom**: Running `petforge quota` without any subcommand should perform
a one-shot probe and print fresh data, per the V3.7 design spec section
"Surface - CLI":

> `petforge quota` (no arg) - One-shot probe + formatted print. Equivalent
> to one daemon tick. Errors with hint if `optIn=false`.

The current `statusCmd` in `src/commands/quota.ts` only reads the cached
quota block from state and formats it - never calls `probe()`. Stale data
is silently displayed.

**Repro**:
```
petforge quota
# Prints whatever the daemon last wrote. May be tens of minutes old.
```

**Suggested fix**:
1. In `src/commands/quota.ts::statusCmd`, after the opt-in guard, resolve
   the token, run `probe()`, and merge via `applyProbeResult()` inside the
   same `withStateLock`. The existing daemon-loop path in the same file is
   the template - just call it once.
2. Update `tests/commands/quota.test.ts` to assert that `petforge quota`
   (default sub) hits `fetchImpl` exactly once and writes a fresh sample.
3. Keep `petforge quota --json` doing the same probe-then-print (output
   format is the only difference).

Add a `--no-probe` flag if Dan wants the old read-only behavior for
scripting. Default behavior matches spec.

## Notes

Both items were detected while live-validating V3.7 against Dan's real
Anthropic OAuth session on 2026-05-21. The V3.7.0 ship is unaffected -
`petforge quota enable` and the daemon path both work correctly. These
are quality-of-life issues to close out before any broader announcement.
