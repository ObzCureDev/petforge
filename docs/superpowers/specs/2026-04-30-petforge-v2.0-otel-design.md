## 1. Background & Context

In April 2026 Anthropic shipped first-class **OpenTelemetry support** for Claude Code, exposing rich, stable, public metrics and event streams. Activated via `CLAUDE_CODE_ENABLE_TELEMETRY=1` plus the standard `OTEL_*` env vars.

The Claude Code OTel surface (as of v2.1.x) includes:

- **Metrics** (`claude_code.*`):
  - `lines_of_code.count` — added / removed (attribute `type`)
  - `token.usage` — input / output / cache_read / cache_creation (attribute `type`, `model`)
  - `cost.usage` — USD
  - `code_edit_tool.decision` — accept / reject (attribute `decision`, `tool`)
  - `tool_decision` — accept / reject (attribute `decision`, `tool_name`)
  - `commit.count`, `pull_request.count`
  - `api.error.count`
  - `session.count`
- **Logs / events** (`claude_code.user_prompt`, `claude_code.tool_result`, `claude_code.api_request`)

PetForge V1.x reads only **hook events** (5 events through `~/.claude/settings.json`), which give per-event firing but no counters for tokens, lines, $ cost, or accept/reject ratios. V2.0 brings these in via OTel.

**Why V2.0 now**: the OTel surface unlocks an entire achievement / stats tier that was simply not measurable from hooks alone. It also keeps PetForge legally clean (consuming a publicly-documented API, not parsing internal Claude files — see §11 of the V1 spec).

**Positioning**: V2.0 is **purely additive**. V1.x users who don't activate OTel keep working as before. Hooks remain canonical for the 50ms-tight feedback loop (XP, level, streak, marathon, polyglot…). OTel adds rich-data counters and 8 new achievements gated on `state.counters.otel.lastUpdate > 0`.

## 2. Goals & Non-Goals

### Goals (V2.0)
- **G1** — Embed a local OTLP/HTTP collector that ingests Claude Code metrics into PetForge state.
- **G2** — Extend the state schema with a `counters.otel` block (cumulative, never reset).
- **G3** — Ship 8 new achievements gated on OTel data.
- **G4** — Provide `petforge init --otel` / `--no-otel` to flip the integration on/off in one command.
- **G5** — Cohabit gracefully with existing OTel collectors (Datadog / Honeycomb / Grafana) via a fan-out env var.
- **G6** — Keep zero-telemetry posture: collector binds to `127.0.0.1` only, no `--lan` flag.
- **G7** — Migrate V1.x state.json transparently (no bump of `schemaVersion`, no breaking change).

### Non-Goals (V2.0)
- **NG1** — Append-only event history (`events.ndjson`). Deferred to V2.1.
- **NG2** — Heatmap visualisation in `petforge serve`. Deferred to V2.1.
- **NG3** — Insights generation ("you code 30% more on Tuesdays"). Deferred to V2.1.
- **NG4** — Cinematic milestones for OTel thresholds. Deferred to V2.2.
- **NG5** — `WISDOM` cosmetic stat derived from cache hit rate. Deferred to V2.2.
- **NG6** — Auto-start as a background service (Windows Service / launchd / systemd). Documented in README only.
- **NG7** — gRPC support. JSON-over-HTTP only — zero protobuf dependency.
- **NG8** — OTel **traces** ingestion. Only metrics + logs surfaces (and even logs only consumed in V2.1).

## 3. Architecture

```
┌─────────────────────────────────────────────────────┐
│ Claude Code v2.1+                                   │
│   ENV: CLAUDE_CODE_ENABLE_TELEMETRY=1               │
│        OTEL_EXPORTER_OTLP_PROTOCOL=http/json        │
│        OTEL_EXPORTER_OTLP_ENDPOINT=http://127...    │
│        OTEL_METRIC_EXPORT_INTERVAL=30000            │
│   Pushes OTLP/HTTP every 30s (configurable)         │
└──────────────┬──────────────────────────────────────┘
               │ POST /v1/metrics
               │ POST /v1/logs (V2.0: ignored, parsed in V2.1)
               ▼
┌─────────────────────────────────────────────────────┐
│ PetForge OTLP collector (petforge collect)          │
│   - HTTP server on 127.0.0.1:7879                   │
│   - Parse OTLP/HTTP JSON                            │
│   - Filter to claude_code.* metric names only       │
│   - Aggregate into state.counters.otel              │
│   - Check OTel-gated achievements                   │
│   - Optional fan-out: PETFORGE_OTEL_FORWARD=URL     │
└──────────────┬──────────────────────────────────────┘
               │ withStateLock (existing)
               ▼
┌─────────────────────────────────────────────────────┐
│ ~/.petforge/state.json (extended schema)            │
│   counters.otel = { ...new fields }                 │
└─────────────────────────────────────────────────────┘
                       ↑ also written by hooks (V1.x, unchanged)
┌─────────────────────────────────────────────────────┐
│ Hooks: 5 Claude Code events (V1 unchanged)          │
└─────────────────────────────────────────────────────┘
```

**Key properties:**
- Hooks and OTel collector are **independent processes** — both lock `state.json` through the existing `proper-lockfile` infrastructure. Concurrent writes serialize cleanly.
- The collector is a **long-running daemon**. Hooks are ephemeral (sub-50ms) and remain the primary canon. OTel updates can lag 30s — that's fine, we never display "live" OTel data, only cumulative counters.
- All OTel ingestion goes through the same `withStateLock(...)` API used by hooks. No new locking primitive.

## 4. State Schema Extension

### 4.1 New `counters.otel` sub-structure

```ts
interface OtelCounters {
  // Cumulative — all-time, never reset
  linesAdded: number;
  linesRemoved: number;
  tokensIn: number;
  tokensOut: number;
  tokensCacheRead: number;
  tokensCacheCreation: number;
  costUsdCents: number;          // stored as cents to avoid float drift
  editsAccepted: number;
  editsRejected: number;
  toolDecisionsAccepted: number;
  toolDecisionsRejected: number;
  commitCount: number;
  prCount: number;
  apiErrorCount: number;

  // Per-model breakdown (cardinality bounded; ~5-10 entries expected)
  modelUsage: Record<string, {
    tokensIn: number;
    tokensOut: number;
    sessions: number;
  }>;

  // Metadata
  lastUpdate: number;            // epoch ms of last successful ingest, 0 = never
  ingesterStarted: number;       // epoch ms of first event ever ingested (used to gate the V1.x → V2.0 migration UX)
}

interface State {
  // ... V1 fields ...
  counters: {
    // ... V1 fields ...
    otel: OtelCounters;          // NEW in V2.0
  };
}
```

### 4.2 Initial values

`createInitialState` and `recoverCorruptState` populate `counters.otel` with all-zero numeric fields, empty `modelUsage`, and `lastUpdate = 0`, `ingesterStarted = 0`.

### 4.3 Migration

`schemaVersion` stays at **1**. The Zod validator marks `counters.otel` as **optional** (`z.object({...}).optional()`). On read:
- If absent: synthesize defaults in-memory and rewrite state.json on next mutation.
- If present: validate normally.

This keeps V1.x state.json files valid and lets V2.0 binaries upgrade them silently on first hook event or first OTel ingest.

## 5. OTLP Collector

### 5.1 HTTP server

- Library: native `node:http`. **No** OpenTelemetry SDK dependency — too heavy for a single-purpose ingester.
- Bind: `127.0.0.1` strict. **No** `--lan` flag (see §11).
- Default port: `7879` (one above `serve`'s 7878 to avoid clash). Override via `--port=N` or env `PETFORGE_OTEL_PORT`.
- Routes:
  - `POST /v1/metrics` — parse + ingest
  - `POST /v1/logs` — accept (HTTP 200) but discard payload (V2.1 will use this)
  - `POST /v1/traces` — reject with HTTP 404 (we don't claim to ingest traces)
  - `GET /healthz` — return `{ "ok": true, "lastUpdate": <epoch> }` for diagnostic
  - Anything else — HTTP 404

### 5.2 Parsing

Accept `Content-Type: application/json` only (Claude Code respects `OTEL_EXPORTER_OTLP_PROTOCOL=http/json`). Reject `application/x-protobuf` with HTTP 415 + a one-line stderr log telling the user to set the protocol env var. **No** protobuf dependency.

Payload limit: **10 MB** hard cap. Above → HTTP 413, drop silently.

OTLP/HTTP/JSON shape (excerpt):

```json
{
  "resourceMetrics": [{
    "resource": { "attributes": [{ "key": "service.name", "value": { "stringValue": "claude-code" } }] },
    "scopeMetrics": [{
      "scope": { "name": "claude_code" },
      "metrics": [{
        "name": "claude_code.lines_of_code.count",
        "sum": {
          "dataPoints": [{
            "asInt": "42",
            "attributes": [{ "key": "type", "value": { "stringValue": "added" } }],
            "timeUnixNano": "1730000000000000000"
          }]
        }
      }]
    }]
  }]
}
```

The collector walks `resourceMetrics[].scopeMetrics[].metrics[]`, filters by `name.startsWith("claude_code.")`, then dispatches to per-metric handlers (see §5.4).

Malformed JSON → HTTP 400, no state mutation, no crash. `PETFORGE_OTEL_DEBUG=1` env var enables verbose stderr logs (received batch summary, parse errors).

### 5.3 Aggregation

OTel metrics in Claude Code are **monotonic Sum** (cumulative). Two aggregation rules:

1. **Cumulative metrics** (lines, tokens, cost, decisions, commits, PRs, errors): the data point's value IS the absolute counter from Claude Code's perspective. We track the **previous value** per (metric_name, attributes) tuple in memory and add only the delta to PetForge counters. This avoids double-counting on Claude Code restarts when counters reset.

   *Tradeoff*: in-memory previous-value state is lost when `petforge collect` restarts. First batch after restart is treated as a fresh baseline (delta = 0). Acceptable: a missed batch every restart is negligible vs. the alternative (persistent ingester state).

2. **Per-model breakdown** (`tokens.usage` with attribute `model`): same logic, indexed by model. Cardinality bounded by Claude's available models (~5-10 entries).

### 5.4 Per-metric handlers

| OTel metric | OTel attribute | PetForge counter |
|---|---|---|
| `claude_code.lines_of_code.count` | `type=added` | `linesAdded` |
| `claude_code.lines_of_code.count` | `type=removed` | `linesRemoved` |
| `claude_code.token.usage` | `type=input` | `tokensIn` + `modelUsage[model].tokensIn` |
| `claude_code.token.usage` | `type=output` | `tokensOut` + `modelUsage[model].tokensOut` |
| `claude_code.token.usage` | `type=cache_read` | `tokensCacheRead` |
| `claude_code.token.usage` | `type=cache_creation` | `tokensCacheCreation` |
| `claude_code.cost.usage` | — | `costUsdCents` (multiply USD by 100, round to int) |
| `claude_code.code_edit_tool.decision` | `decision=accept` | `editsAccepted` |
| `claude_code.code_edit_tool.decision` | `decision=reject` | `editsRejected` |
| `claude_code.tool_decision` | `decision=accept` | `toolDecisionsAccepted` |
| `claude_code.tool_decision` | `decision=reject` | `toolDecisionsRejected` |
| `claude_code.commit.count` | — | `commitCount` |
| `claude_code.pull_request.count` | — | `prCount` |
| `claude_code.api.error.count` | — | `apiErrorCount` |
| `claude_code.session.count` | — | (unused; we track sessions via hooks) |
| Anything else | — | ignored |

Unknown metrics or unknown attribute values are silently ignored — forward-compatible.

After ingestion, `lastUpdate = Date.now()`, `ingesterStarted` set on first non-zero ingest, then `checkOtelAchievements(state, now)` runs.

### 5.5 Fan-out

Env var `PETFORGE_OTEL_FORWARD=http://...` (or flag `--forward=URL`, flag overrides env) — after ingesting locally, the collector POSTs the original raw body to the forward URL, fire-and-forget, with a 1s timeout. Errors are logged to stderr only (when `PETFORGE_OTEL_DEBUG=1`), never affect ingest.

This lets users keep Datadog/Honeycomb/Grafana while still feeding PetForge.

### 5.6 Lifecycle

- `petforge collect` — foreground daemon. Logs nothing on stdout (silent like hooks) unless `PETFORGE_OTEL_DEBUG=1` is set. Logs `petforge collector listening on http://127.0.0.1:7879` on stderr at startup.
- Clean shutdown on SIGINT/SIGTERM: stop accepting new connections, flush in-flight requests, release lock.
- No auto-restart in V2.0. Documented as a manual concern (see README troubleshooting).

## 6. Achievements V2.0

### 6.1 Eight new achievements

| ID | Name | Trigger | XP | Tier |
|---|---|---|---|---|
| `code_architect` | Code Architect | `linesAdded ≥ 10 000` | 3 000 | Hard |
| `code_titan` | Code Titan | `linesAdded ≥ 100 000` | 10 000 | Mythic |
| `token_whisperer_v2` | Token Whisperer ⚡ | `tokensIn + tokensOut ≥ 1 000 000` | 3 000 | Hard |
| `cache_lord` | Cache Lord | `tokensCacheRead / (tokensIn + tokensCacheRead) ≥ 0.80` AND `tokensIn + tokensCacheRead ≥ 100 000` | 2 500 | Medium |
| `frugal_coder` | Frugal Coder | `promptsTotal ≥ 100` AND `costUsdCents ≤ 100` | 1 500 | Easy |
| `big_spender` | Big Spender | `costUsdCents ≥ 10 000` ($100 cumulative) | 2 000 | Medium |
| `pr_machine` | PR Machine | `prCount ≥ 50` | 3 000 | Hard |
| `picky_reviewer` | Picky Reviewer | `editsRejected ≥ 50` | 1 500 | Medium |

The V1 `tool_whisperer` (1000 tool uses) **stays as-is**. The new one is `token_whisperer_v2` with the ⚡ glyph to disambiguate visually. We do not rename V1 to keep state.json `unlocked` arrays valid across versions.

### 6.2 Gating

All 8 achievements gate on `state.counters.otel.lastUpdate > 0`. Without that, the user has never run the collector — there's no OTel data, and these achievements should remain invisible / never fire from stale-zero counters.

### 6.3 Detection point

After every successful OTLP ingest, `checkOtelAchievements(state, now)` runs **inside the same `withStateLock(...)` call** — so achievements unlock atomically with the counter update. Same pattern as V1's `checkAchievementsForEvent` for hooks.

Newly unlocked IDs go into `state.achievements.pendingUnlocks` (existing field). Cinematics replay through the existing `petforge` / `petforge card` flow on next display — zero render changes.

### 6.4 XP from achievements

Same as V1: unlock adds `def.xp` to `state.progress.xp`. After ingest, level/phase recomputation runs (same code path as `applyHookEvent`). If a centurion-level threshold is crossed via OTel-driven XP, that's fine — `centurion` is already checked on every level recomputation.

## 7. CLI Commands

### 7.1 `petforge collect`

New top-level command. Foreground daemon.

```
petforge collect [--port=7879] [--forward=URL]

Run the OTLP/HTTP collector. Listens on 127.0.0.1:<port> for Claude Code
metrics and ingests them into ~/.petforge/state.json.

Options:
  --port=N        Override default port (7879). Env: PETFORGE_OTEL_PORT
  --forward=URL   POST received bodies to URL after local ingest.
                  Env: PETFORGE_OTEL_FORWARD

Press Ctrl+C to stop.
```

Exit codes:
- `0` — clean shutdown
- `1` — port in use, payload limit error setup, etc.

### 7.2 `petforge init --otel` / `--no-otel`

Extends the existing V1 command. Flags are exclusive:
- `--otel` adds the OTel env block to `~/.claude/settings.json`'s `env` field.
- `--no-otel` strips that block (leaves `state.counters.otel` data intact).
- Without either flag: existing V1 behavior (just hooks).

The env block written:

```json
{
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
    "OTEL_METRICS_EXPORTER": "otlp",
    "OTEL_LOGS_EXPORTER": "otlp",
    "OTEL_EXPORTER_OTLP_PROTOCOL": "http/json",
    "OTEL_EXPORTER_OTLP_ENDPOINT": "http://127.0.0.1:7879",
    "OTEL_METRIC_EXPORT_INTERVAL": "30000"
  }
}
```

If the user has manually set any of these env vars to non-PetForge values, `petforge init --otel` warns and aborts unless `--force` is passed.

Backup as usual (`.bak` / timestamped).

### 7.3 `petforge doctor` updates

Add three new diagnostic checks (warnings, not critical):
- OTel env vars present in `~/.claude/settings.json`
- Collector is reachable on `http://127.0.0.1:7879/healthz`
- `state.counters.otel.lastUpdate` is within the last 24h (warn: "no OTel ingest in 24h, did `petforge collect` stop?")

## 8. Display Additions

### 8.1 `petforge card`

A second activity line appears **only if** `state.counters.otel.lastUpdate > 0`:

```
Sessions: 12 · Streak: 3d · Prompts: 47 · Tools: 312
Lines: +8 234 / -1 109 · Tokens: 1.2M · Cost: $4.30 · Cache: 78%
```

If `lastUpdate === 0` → the OTel line is omitted. Zero noise for V1.x users.

Format details:
- Lines: thousand-separator, leading +/-
- Tokens: K/M/B suffix (Intl.NumberFormat compact mode)
- Cost: $X.XX, computed `costUsdCents / 100`
- Cache: percentage, `tokensCacheRead / (tokensIn + tokensCacheRead) × 100`, rounded to int

### 8.2 `petforge serve` (web view)

Exact same logic — OTel line shown conditionally. The activity div in the HTML page gets a sub-div `id="otel-activity"` populated from `state.counters.otel`. SSE pushes both blocks together in the existing JSON envelope.

### 8.3 `petforge watch`

Same — the existing `ActivityBlock` component receives the OTel sub-line conditionally.

The `ActivityBlock` component becomes the single source of truth for activity rendering across all three surfaces (`card`, `serve`, `watch`).

## 10. Migration V1.x → V2.0

Three migration paths:

1. **Cold V2.0 install on a V1.x state.json**: state.counters.otel auto-synthesized with zeros on first read. Subsequent hook write persists the new field. No collector activity required — V1 features keep working.
2. **`petforge init --otel` on V1.x**: patches settings.json (adds env block, keeps hooks). User must launch `petforge collect` manually. First successful ingest sets `ingesterStarted` and `lastUpdate`.
3. **Reset (`rm ~/.petforge/state.json`)**: V2.0 binary creates fresh state with empty `counters.otel`. Same as cold V2.0 install on a fresh machine.

`schemaVersion` stays at `1`. We do **not** bump because:
- Zod schema accepts old (no `otel`) and new (with `otel`) shapes.
- Hooks code never assumes `counters.otel` is set; it only reads counters it owns (V1 fields).
- Future schema bump (V3+) can rationalize this.

A clear pattern for the future: as long as new fields are non-required and have safe defaults, no schema version bump.

## 11. Security Model

PetForge V1's core promise — **zero telemetry, zero phone-home, no Anthropic file copying** — is preserved. V2.0 adds three concrete rules:

- ❌ **Collector NEVER binds to non-loopback**. No `--lan` flag. The OTel payload from Claude Code carries truncated user prompts, file paths, and cost signals — exposing this on a LAN is a privacy hazard. Multi-machine setups: user puts a reverse proxy with auth in front. Out of scope.
- ❌ **Collector never reads internal Claude files**. Same as V1's Buddy rule. We consume only the public OTel push surface that Claude Code voluntarily exports.
- ❌ **No persistence of raw payloads** in V2.0. The body is parsed, aggregated to counters, and discarded. (V2.1 will add `events.ndjson` with explicit user opt-in.)
- ✅ **Fan-out**, when configured, sends the **exact original body** the collector received. PetForge does not re-serialize or alter it — we trust users to know where they're forwarding to.
- ✅ **Health check `/healthz`** does not expose any state data — only `{ ok: true, lastUpdate: <epoch> }`.

## 12. Risks

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| Anthropic changes the OTel surface | Medium | Medium | We filter by metric name; unknown metrics ignored. New metrics added by Anthropic produce no error, just no PetForge feature. |
| Counter "reset" on Claude Code restart double-counts | Medium | Low | Per-(metric,attrs) previous-value memo ensures we only add deltas. On collector restart we lose memo (= miss one batch on rare restart pairs). Acceptable. |
| User has a non-default `OTEL_*` env stack | Low | Medium | `init --otel` warns + aborts unless `--force`. Forward env var lets advanced users chain collectors. |
| OTLP/JSON payload too large | Low | Medium | 10 MB cap. Drop silently → HTTP 413. Collector survives, no state corruption. |
| Concurrent hook write + OTel ingest | High | Low | Both go through `withStateLock(...)`. Existing infra handles serialisation. |
| Port 7879 already in use | Medium | Low | Collector exits 1 with an explicit error message + suggests `--port=N`. Doctor flags it. |
| User runs `init --otel` but never `collect` | High | Low | Doctor flags `lastUpdate` warning. Counters stay at zero. OTel achievements simply never fire. |

## 14. Tests (priority order)

1. **Collector — parsing**
   - Valid OTLP/JSON metrics payload → counters incremented correctly
   - Mixed `claude_code.*` and other metrics → only `claude_code.*` consumed
   - Malformed JSON → HTTP 400, no state mutation, no crash
   - Payload > 10 MB → HTTP 413, drop silent
   - Wrong Content-Type (protobuf) → HTTP 415 + helpful stderr message
2. **Collector — aggregation**
   - Cumulative delta logic: two batches with growing values → only the delta is added
   - Counter "reset" simulation (lower value than memo) → first batch becomes new baseline
   - Per-model breakdown: distinct models tracked independently
3. **Achievements**
   - Each of the 8 fires exactly once at threshold
   - Gating: with `lastUpdate === 0`, no OTel achievement can fire even if counters are at threshold (defensive)
   - `code_architect` and `code_titan` both fire when crossing 100K
4. **Schema migration**
   - V1.x state.json (no `counters.otel`) is read silently and upgraded on next write
   - Round-trip (read → write → read) preserves all fields
5. **`petforge init`**
   - `--otel` writes the env block, leaves hooks intact
   - `--no-otel` strips the env block, leaves hooks + counters intact
   - Conflict detection: pre-existing `OTEL_*` env in settings.json triggers `--force` requirement
6. **Fan-out**
   - With `PETFORGE_OTEL_FORWARD=URL` set, the collector POSTs the original body to URL
   - Fan-out failure does not affect local ingest
7. **Health check**
   - `GET /healthz` returns `{ ok: true, lastUpdate }` without state mutation
8. **Doctor**
   - Three new checks (env vars present, collector reachable, recent ingest) emit correct status
